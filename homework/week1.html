Q1: For the three categories of web software technology we've just defined, you are asked to find out which category so-called "Netscape plug-in" technology belongs to. Please describe the basic idea of the technology, and give your reasoning on its position in the technology categories.

    I think the so-called "Netscape plug-in" technology belongs to the distributed object technologies.
    Netscape plug-in is a kind of browser extension that run on Netscape Plugin Application Programming Interface (NPAPI). It was first developed for Netscape browsers, starting in 1995 with Netscape Navigator 2.0, but was subsequently adopted by other browsers. It helps the browser to explain contents that the browser can’t handle natively.
    According to WIKI, “Netscape plug-in declares content types it can handle in NPAPI architecture. When the browser encounters a content type it cannot handle natively, it loads the appropriate plugin, sets aside space within the browser context for the plugin to render and then streams data to it. The plugin is responsible for rendering the data. The plugin runs in-place within the page, as opposed to older browsers that had to launch an external application to handle unknown content types.”
    My reason on its position is that Netscape plug-in sits at browser side but it’s responsible for handling data from the server, which fits the definition of "distributed object technology”. The plug-in is developed to interplay with the content or object on different addresses. It’s similar with ActiveX mentioned in the course.



Q2: In a normal intercommunication between Web browser and Web Server, Web Server closes the network communication right after it sends back the document requested by the browser. Think about this statusless character of World Wide Web. Give some good points and bad points of it. Can a web server know exactly how many browsers are reading its document?

Good points:
1. It’s a efficient way to make full use of the limited resources of the Web Server. The act of closing the network communication frees the occupied ports, so that the Server can make more connections.
2. It simplify the server design because there is no need to dynamically allocate storage to deal with conversations in progress.

Bad points:
1. Sometimes it can be inefficient when the Client asks for resources continuously. The Client will have to reconnect the Server every time it needs resources, which may take a lot of time.
2. It may be necessary to include additional information in every request, and this extra information will need to be interpreted by the server.

    For the last question, I think the server can’t know exactly how many browsers are reading its document when the connections haven’t reached the limit. When the connections reach the limit of the server, apparently, the number of browsers which are reading its document is the max number of links it can hold.
